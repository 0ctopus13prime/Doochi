599 /*
600  The 'zlib' compression library provides in-memory compression and
601  decompression functions, including integrity checks of the uncompressed data.
602  This version of the library supports only one compression method (deflation)
603  but other algorithms will be added later and will have the same stream
604  interface.
605  Compression can be done in a single step if the buffers are large enough,
606  or can be done by repeated calls of the compression function. In the latter
607  case, the application must provide more input and/or consume the output
608  (providing more output space) before each call.
609  The compressed data format used by default by the in-memory functions is
610  the zlib format, which is a zlib wrapper documented in RFC 1950, wrapped
611  around a deflate stream, which is itself documented in RFC 1951.
612  The library also supports reading and writing files in gzip (.gz) format
613  with an interface similar to that of stdio using the functions that start
614  with "gz". The gzip format is different from the zlib format. gzip is a
615  gzip wrapper, documented in RFC 1952, wrapped around a deflate stream.
616  This library can optionally read and write gzip and raw deflate streams in
617  memory as well.
618  The zlib format was designed to be compact and fast for use in memory
619  and on communications channels. The gzip format was designed for single-
620  file compression on file systems, has a larger header than zlib to maintain
621  directory information, and uses a different, slower check method than zlib.
622  The library does not install any signal handler. The decoder checks
623  the consistency of the compressed data, so the library should never crash
624  even in the case of corrupted input.
625 */

789 ZEXTERN int ZEXPORT deflate OF((z_streamp strm, int flush));
790 /*
791  deflate compresses as much data as possible, and stops when the input
792  buffer becomes empty or the output buffer becomes full. It may introduce
793  some output latency (reading input without producing any output) except when
794  forced to flush.
795  The detailed semantics are as follows. deflate performs one or both of the
796  following actions:
797  - Compress more input starting at next_in and update next_in and avail_in
798  accordingly. If not all input can be processed (because there is not
799  enough room in the output buffer), next_in and avail_in are updated and
800  processing will resume at this point for the next call of deflate().
801  - Generate more output starting at next_out and update next_out and avail_out
802  accordingly. This action is forced if the parameter flush is non zero.
803  Forcing flush frequently degrades the compression ratio, so this parameter
804  should be set only when necessary. Some output may be provided even if
805  flush is zero.
806  Before the call of deflate(), the application should ensure that at least
807  one of the actions is possible, by providing more input and/or consuming more
808  output, and updating avail_in or avail_out accordingly; avail_out should
809  never be zero before the call. The application can consume the compressed
810  output when it wants, for example when the output buffer is full (avail_out
811  == 0), or after each call of deflate(). If deflate returns Z_OK and with
812  zero avail_out, it must be called again after making room in the output
813  buffer because there might be more output pending. See deflatePending(),
814  which can be used if desired to determine whether or not there is more ouput
815  in that case.
816  Normally the parameter flush is set to Z_NO_FLUSH, which allows deflate to
817  decide how much data to accumulate before producing output, in order to
818  maximize compression.
819  If the parameter flush is set to Z_SYNC_FLUSH, all pending output is
820  flushed to the output buffer and the output is aligned on a byte boundary, so
821  that the decompressor can get all input data available so far. (In
822  particular avail_in is zero after the call if enough output space has been
823  provided before the call.) Flushing may degrade compression for some
824  compression algorithms and so it should be used only when necessary. This
825  completes the current deflate block and follows it with an empty stored block
826  that is three bits plus filler bits to the next byte, followed by four bytes
827  (00 00 ff ff).
828  If flush is set to Z_PARTIAL_FLUSH, all pending output is flushed to the
829  output buffer, but the output is not aligned to a byte boundary. All of the
830  input data so far will be available to the decompressor, as for Z_SYNC_FLUSH.
831  This completes the current deflate block and follows it with an empty fixed
832  codes block that is 10 bits long. This assures that enough bytes are output
833  in order for the decompressor to finish the block before the empty fixed
834  codes block.
835  If flush is set to Z_BLOCK, a deflate block is completed and emitted, as
836  for Z_SYNC_FLUSH, but the output is not aligned on a byte boundary, and up to
837  seven bits of the current block are held to be written as the next byte after
838  the next deflate block is completed. In this case, the decompressor may not
839  be provided enough bits at this point in order to complete decompression of
840  the data provided so far to the compressor. It may need to wait for the next
841  block to be emitted. This is for advanced applications that need to control
842  the emission of deflate blocks.
843  If flush is set to Z_FULL_FLUSH, all output is flushed as with
844  Z_SYNC_FLUSH, and the compression state is reset so that decompression can
845  restart from this point if previous compressed data has been damaged or if
846  random access is desired. Using Z_FULL_FLUSH too often can seriously degrade
847  compression.
848  If deflate returns with avail_out == 0, this function must be called again
849  with the same value of the flush parameter and more output space (updated
850  avail_out), until the flush is complete (deflate returns with non-zero
851  avail_out). In the case of a Z_FULL_FLUSH or Z_SYNC_FLUSH, make sure that
852  avail_out is greater than six to avoid repeated flush markers due to
853  avail_out == 0 on return.
854  If the parameter flush is set to Z_FINISH, pending input is processed,
855  pending output is flushed and deflate returns with Z_STREAM_END if there was
856  enough output space. If deflate returns with Z_OK or Z_BUF_ERROR, this
857  function must be called again with Z_FINISH and more output space (updated
858  avail_out) but no more input data, until it returns with Z_STREAM_END or an
859  error. After deflate has returned Z_STREAM_END, the only possible operations
860  on the stream are deflateReset or deflateEnd.
861  Z_FINISH can be used in the first deflate call after deflateInit if all the
862  compression is to be done in a single step. In order to complete in one
863  call, avail_out must be at least the value returned by deflateBound (see
864  below). Then deflate is guaranteed to return Z_STREAM_END. If not enough
865  output space is provided, deflate will not return Z_STREAM_END, and it must
866  be called again as described above.
867  deflate() sets strm->adler to the Adler-32 checksum of all input read
868  so far (that is, total_in bytes). If a gzip stream is being generated, then
869  strm->adler will be the CRC-32 checksum of the input read so far. (See
870  deflateInit2 below.)
871  deflate() may update strm->data_type if it can make a good guess about
872  the input data type (Z_BINARY or Z_TEXT). If in doubt, the data is
873  considered binary. This field is only for information purposes and does not
874  affect the compression algorithm in any manner.
875  deflate() returns Z_OK if some progress has been made (more input
876  processed or more output produced), Z_STREAM_END if all input has been
877  consumed and all output has been produced (only when flush is set to
878  Z_FINISH), Z_STREAM_ERROR if the stream state was inconsistent (for example
879  if next_in or next_out was Z_NULL or the state was inadvertently written over
880  by the application), or Z_BUF_ERROR if no progress is possible (for example
881  avail_in or avail_out was zero). Note that Z_BUF_ERROR is not fatal, and
882  deflate() can be called again with more input and more output space to
883  continue compressing.
884 */

921 ZEXTERN int ZEXPORT inflate OF((z_streamp strm, int flush));
922 /*
923  inflate decompresses as much data as possible, and stops when the input
924  buffer becomes empty or the output buffer becomes full. It may introduce
925  some output latency (reading input without producing any output) except when
926  forced to flush.
927  The detailed semantics are as follows. inflate performs one or both of the
928  following actions:
929  - Decompress more input starting at next_in and update next_in and avail_in
930  accordingly. If not all input can be processed (because there is not
931  enough room in the output buffer), then next_in and avail_in are updated
932  accordingly, and processing will resume at this point for the next call of
933  inflate().
934  - Generate more output starting at next_out and update next_out and avail_out
935  accordingly. inflate() provides as much output as possible, until there is
936  no more input data or no more space in the output buffer (see below about
937  the flush parameter).
938  Before the call of inflate(), the application should ensure that at least
939  one of the actions is possible, by providing more input and/or consuming more
940  output, and updating the next_* and avail_* values accordingly. If the
941  caller of inflate() does not provide both available input and available
942  output space, it is possible that there will be no progress made. The
943  application can consume the uncompressed output when it wants, for example
944  when the output buffer is full (avail_out == 0), or after each call of
945  inflate(). If inflate returns Z_OK and with zero avail_out, it must be
946  called again after making room in the output buffer because there might be
947  more output pending.
948  The flush parameter of inflate() can be Z_NO_FLUSH, Z_SYNC_FLUSH, Z_FINISH,
949  Z_BLOCK, or Z_TREES. Z_SYNC_FLUSH requests that inflate() flush as much
950  output as possible to the output buffer. Z_BLOCK requests that inflate()
951  stop if and when it gets to the next deflate block boundary. When decoding
952  the zlib or gzip format, this will cause inflate() to return immediately
953  after the header and before the first block. When doing a raw inflate,
954  inflate() will go ahead and process the first block, and will return when it
955  gets to the end of that block, or when it runs out of data.
956  The Z_BLOCK option assists in appending to or combining deflate streams.
957  To assist in this, on return inflate() always sets strm->data_type to the
958  number of unused bits in the last byte taken from strm->next_in, plus 64 if
959  inflate() is currently decoding the last block in the deflate stream, plus
960  128 if inflate() returned immediately after decoding an end-of-block code or
961  decoding the complete header up to just before the first byte of the deflate
962  stream. The end-of-block will not be indicated until all of the uncompressed
963  data from that block has been written to strm->next_out. The number of
964  unused bits may in general be greater than seven, except when bit 7 of
965  data_type is set, in which case the number of unused bits will be less than
966  eight. data_type is set as noted here every time inflate() returns for all
967  flush options, and so can be used to determine the amount of currently
968  consumed input in bits.
969  The Z_TREES option behaves as Z_BLOCK does, but it also returns when the
970  end of each deflate block header is reached, before any actual data in that
971  block is decoded. This allows the caller to determine the length of the
972  deflate block header for later use in random access within a deflate block.
973  256 is added to the value of strm->data_type when inflate() returns
974  immediately after reaching the end of the deflate block header.
975  inflate() should normally be called until it returns Z_STREAM_END or an
976  error. However if all decompression is to be performed in a single step (a
977  single call of inflate), the parameter flush should be set to Z_FINISH. In
978  this case all pending input is processed and all pending output is flushed;
979  avail_out must be large enough to hold all of the uncompressed data for the
980  operation to complete. (The size of the uncompressed data may have been
981  saved by the compressor for this purpose.) The use of Z_FINISH is not
982  required to perform an inflation in one step. However it may be used to
983  inform inflate that a faster approach can be used for the single inflate()
984  call. Z_FINISH also informs inflate to not maintain a sliding window if the
985  stream completes, which reduces inflate's memory footprint. If the stream
986  does not complete, either because not all of the stream is provided or not
987  enough output space is provided, then a sliding window will be allocated and
988  inflate() can be called again to continue the operation as if Z_NO_FLUSH had
989  been used.
990  In this implementation, inflate() always flushes as much output as
991  possible to the output buffer, and always uses the faster approach on the
992  first call. So the effects of the flush parameter in this implementation are
993  on the return value of inflate() as noted below, when inflate() returns early
994  when Z_BLOCK or Z_TREES is used, and when inflate() avoids the allocation of
995  memory for a sliding window when Z_FINISH is used.
996  If a preset dictionary is needed after this call (see inflateSetDictionary
997  below), inflate sets strm->adler to the Adler-32 checksum of the dictionary
998  chosen by the compressor and returns Z_NEED_DICT; otherwise it sets
999  strm->adler to the Adler-32 checksum of all output produced so far (that is,
1000  total_out bytes) and returns Z_OK, Z_STREAM_END or an error code as described
1001  below. At the end of the stream, inflate() checks that its computed Adler-32
1002  checksum is equal to that saved by the compressor and returns Z_STREAM_END
1003  only if the checksum is correct.
1004  inflate() can decompress and check either zlib-wrapped or gzip-wrapped
1005  deflate data. The header type is detected automatically, if requested when
1006  initializing with inflateInit2(). Any information contained in the gzip
1007  header is not retained unless inflateGetHeader() is used. When processing
1008  gzip-wrapped deflate data, strm->adler32 is set to the CRC-32 of the output
1009  produced so far. The CRC-32 is checked against the gzip trailer, as is the
1010  uncompressed length, modulo 2^32.
1011  inflate() returns Z_OK if some progress has been made (more input processed
1012  or more output produced), Z_STREAM_END if the end of the compressed data has
1013  been reached and all uncompressed output has been produced, Z_NEED_DICT if a
1014  preset dictionary is needed at this point, Z_DATA_ERROR if the input data was
1015  corrupted (input stream not conforming to the zlib format or incorrect check
1016  value, in which case strm->msg points to a string with a more specific
1017  error), Z_STREAM_ERROR if the stream structure was inconsistent (for example
1018  next_in or next_out was Z_NULL, or the state was inadvertently written over
1019  by the application), Z_MEM_ERROR if there was not enough memory, Z_BUF_ERROR
1020  if no progress was possible or if there was not enough room in the output
1021  buffer when Z_FINISH is used. Note that Z_BUF_ERROR is not fatal, and
1022  inflate() can be called again with more input and more output space to
1023  continue decompressing. If Z_DATA_ERROR is returned, the application may
1024  then call inflateSync() to look for a good compression block if a partial
1025  recovery of the data is to be attempted.
1026 */

1114  without producing any compressed output. When using the zlib format, this
1115  function must be called immediately after deflateInit, deflateInit2 or
